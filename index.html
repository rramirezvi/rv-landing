Optimización de Query (PostgreSQL)

Reto: Consulta de reportes tardaba ~5s.

  • Se analizó plan con EXPLAIN (ANALYZE, BUFFERS)
  • Índices compuestos + partición por mes
-- Índice compuesto clave
CREATE INDEX CONCURRENTLY idx_gestiones_deudor_fecha
ON lc_gestiones (id_deudor, fecha_gestion DESC)
INCLUDE (id_tipo_resultado);

-- Ejemplo de hint lógico: limitar rango por fecha
SELECT ...
FROM lc_gestiones
WHERE fecha_gestion >= date_trunc('month', now())::date
  AND fecha_gestion < (date_trunc('month', now()) + INTERVAL '1 month')::date;

📉 Resultado: 5s → 180ms (×27.7 más rápido).

Eficiencia de Pagos (PL/pgSQL)

Reto: KPIs M/M y pivots por día/gestor/segmento.

CREATE OR REPLACE FUNCTION eficiencia_mes(mes int, anio int)
RETURNS TABLE(gestor text, meta numeric, recupero numeric, eficiencia text) AS $$
BEGIN
  RETURN QUERY
  SELECT g.nombre, m.meta_1, SUM(p.monto),
         ROUND(SUM(p.monto)*100/m.meta_1, 2) || '%'
  FROM metas m
  JOIN pagos p ON p.fecha BETWEEN m.fecha_inicio AND m.fecha_cierre
  JOIN gestores g ON g.id = p.id_gestor
  GROUP BY g.nombre, m.meta_1;
END; $$ LANGUAGE plpgsql;

📈 Resultado: Métricas consistentes en BD, sin depender del front.
